# 【開發策畫說明書】多世界觀動態地圖與角色排程系統

## 一、 系統總覽 (System Overview)
本系統基於「NEXUS PARALLAX 視差世界觀」架構，要求系統具備**「程序化生成 (Procedural Generation)」**能力。系統不預先寫死地圖與排程，而是採用「兩階段 AI 生成流水線」：
1. **第一階段 (Map Gen)**：讀取當前「世界書 (Worldbook)」，由 AI 動態生成該世界的區域與設施清單 (JSON)。
2. **第二階段 (Schedule Gen)**：將「生成好的地圖清單」與「角色卡」傳入，由 AI 為角色生成帶有機率分支的一日作息表 (JSON)。

---

## 二、 第一階段：動態世界地圖生成 (Dynamic Map Generation)

### 1. 觸發時機與邏輯
當玩家切換或創建新世界觀時，系統提取當前【世界書設定】，呼叫 AI 生成地圖。

### 2. 嚴格生成規則
*   **認知框架隔離**：設施的名稱、性質必須嚴格符合該世界觀。例如：中古世界不可生成「捷運站」或「網咖」。
*   **層級結構**：世界 (World) -> 區域 (Zone) -> 設施 (Facility)。
*   **數量限制**：生成 2~3 個區域，每個區域包含 3~5 個設施。

### 3. 輸出資料結構 (JSON)
AI 必須嚴格輸出以下 JSON 格式，供系統寫入全域變數（如 `win.CURRENT_WORLD_DATA`）：
```json
{
  "world_name": "艾爾迪亞王國 (範例)",
  "zones": [
    {
      "zone_id": "Zone_A",
      "zone_name": "王都中心",
      "facilities": [
        { "id": "A_Castle", "name": "皇家城堡", "icon": "🏰" },
        { "id": "A_Tavern", "name": "躍馬旅店", "icon": "🍻" }
      ]
    }
  ]
}
```
*(注意：此處生成的 `id` 將作為第二階段排程綁定的唯一識別碼)*

---

## 三、 第二階段：動態角色排程生成 (Dynamic Schedule Generation)

### 1. 觸發時機與邏輯
地圖生成完畢後，系統遍歷該世界的登場角色，將【世界書】、【地圖 JSON】與【角色資訊】傳入 AI，生成該角色的排程狀態機。

### 2. 嚴格生成規則 (核心重點)
這部分邏輯必須嚴密，不可有漏，請 Claude 嚴格實作此資料結構的解析與應用：

*   **時段劃分 (Time Periods)**：必須且只能包含以下 7 個時段，覆蓋 24 小時。
    *   `黎明` (05:00-08:00)
    *   `上午` (08:00-12:00)
    *   `下午` (12:00-17:00)
    *   `黃昏` (17:00-19:00)
    *   `晚上` (19:00-23:00)
    *   `午夜` (23:00-02:00)
    *   `凌晨` (02:00-05:00)
*   **分支與機率 (Probability Branches)**：
    *   每個時段最多設定 **3 條**分支路線。
    *   該時段內所有分支的 `prob` (機率) 數值加總**必須剛好等於 100**。
*   **活動分類 (Category)**：嚴格歸類為以下三種：
    1.  `Work` (工作類)：上班、開會、出任務等。
    2.  `Home` (宅家類)：睡覺、休息等。
    3.  `Outing` (外出類)：吃飯、娛樂、散步等。
*   **地點綁定 (Location Binding)**：
    *   若為 `Work` 或 `Outing`，`location_id` **必須**從「第一階段生成的地圖 JSON」中挑選對應的設施 ID (如 `A_Tavern`)。
    *   若為 `Home`，`location_id` 固定填寫為 `"Home"`。
*   **行為與台詞 (Action & Dialogue)**：
    *   必須包含符合該世界觀認知框架的動作 (`action`) 與台詞 (`dialogue`)，不可出現破壞沉浸感的現代詞彙。

### 3. 輸出資料結構 (JSON)
```json
{
  "char_id": "角色識別碼",
  "schedule": {
    "下午": [
      {
        "category": "Outing",
        "prob": 30,
        "location_id": "A_Tavern",
        "action": "坐在角落擦拭著長劍",
        "dialogue": "這點賞金連買杯好麥酒都不夠..."
      },
      {
        "category": "Home",
        "prob": 70,
        "location_id": "Home",
        "action": "躺在簡陋的木床上呼呼大睡",
        "dialogue": "Zzz..."
      }
    ]
    // ... 必須補齊其餘 6 個時段
  }
}
```

## 四、 給開發者 (Claude) 的實作要求

請 Claude 基於以上策畫，幫我編寫/重構以下 JavaScript 功能模組：

1.  **地圖數據管理器 (Map Data Manager)**：
    *   需要一個函數接收第一階段的 JSON，並動態構建前端 UI（產生區域按鈕與設施網格）。
2.  **排程解析與骰子引擎 (Schedule Engine & Dice Roller)**：
    *   編寫一個 `getCurrentActivity(char_id, currentTime)` 函數。
    *   **邏輯**：判斷 `currentTime` 屬於哪一個時段 (如：下午) -> 讀取該角色該時段的 Array -> **產生 1~100 的隨機亂數 (Dice)** -> 根據 `prob` 權重分配，回傳最終選中的那條分支物件。
3.  **地圖小人渲染綁定 (Renderer Integration)**：
    *   當玩家點擊進入某個設施 (`location_id`) 時，遍歷所有角色的 `getCurrentActivity()` 結果。
    *   如果回傳結果的 `location_id` 等於玩家當前所在的設施 ID，則將該角色的 `action` 與 `dialogue` 渲染為地圖上的動態小人（顯示頭像與台詞氣泡）。
